# 1. システム概要

`shared_shop`は、プログラミング研修用の会員制ECサイトです。一般ユーザー向けの購入機能と、運用管理者・システム管理者向けの管理機能を備えています。

## 技術構成

- Java 17
- Spring Boot 4.0.6
- Spring MVC
- Spring Data JPA
- Thymeleaf
- Oracle Database（ojdbc11）
- Jakarta Bean Validation
- Maven Wrapper
- HTML / CSS / JavaScript

## 実行時の基本設定

- アプリケーション本体：`shared_shop/`
- コンテキストパス：`/shared_shop`
- 標準ポート：`55000`
- ローカルURL例：`http://localhost:55000/shared_shop/`
- DB接続：Oracle 21cの`xepdb1`、または環境に応じてOracle 11gの`xe`
- セッション内のログイン情報：属性名`user`、型`UserBean`

`application.properties`にはローカル環境固有のDB接続情報が含まれます。認証情報や個人環境の設定をIssueの目的なく変更・コミットしないでください。

# 2. アーキテクチャとディレクトリ

本プロジェクトは、ControllerからSpring Data JPAのRepositoryを直接呼び出す、研修用の基本的なMVC構成です。

```text
shared_shop/
├─ src/main/java/jp/co/sss/shop/
│  ├─ annotation/   独自Bean Validationアノテーション
│  ├─ bean/         画面・セッション受け渡し用Bean
│  ├─ config/       Filter等の登録設定
│  ├─ controller/   MVC Controller
│  ├─ entity/       JPA Entity
│  ├─ filter/       認証・権限・共通画面情報のFilter
│  ├─ form/         入力Form
│  ├─ repository/   Spring Data JPA Repository
│  ├─ service/      既存の変換・計算・ファイル補助クラス
│  ├─ util/         定数・URL判定
│  └─ validator/    独自Bean Validation
├─ src/main/resources/
│  ├─ templates/    Thymeleafテンプレート
│  ├─ static/css/   CSS
│  ├─ static/js/    JavaScript
│  ├─ messages.properties
│  └─ ValidationMessages.properties
└─ src/test/java/   JUnit / Mockitoテスト
```

既存の`service`パッケージには`BeanTools`、`PriceCalc`、`UploadFileService`がありますが、一般的な業務ロジックのサービスレイヤではありません。追加機能のために新しいサービスレイヤを導入せず、既存構成に合わせてController、Repository、補助クラスへ責務を分けてください。

# 4. データモデル

フェーズ1時点の主要テーブルは以下の5つです。

| テーブル | 内容 | 主な関係 |
| --- | --- | --- |
| `users` | 会員情報 | 1ユーザーに複数注文 |
| `categories` | 商品カテゴリ | 1カテゴリに複数商品 |
| `items` | 商品情報 | カテゴリ、注文明細に関連 |
| `orders` | 注文情報 | ユーザー、複数注文明細に関連 |
| `order_items` | 注文明細 | 注文と商品に関連 |

```text
User 1 ── * Order 1 ── * OrderItem * ── 1 Item * ── 1 Category
```

## 既存データ設計上の注意

- 主キーはOracle Sequenceを利用し、Entityでは`@SequenceGenerator`を使用します。
- `users`、`categories`、`items`は`delete_flag`による論理削除を前提とします。
- 通常表示・認証・更新では`Constant.NOT_DELETED`を条件に含めてください。
- 注文履歴は会員退会後も保持する設計です。関連データを安易に物理削除しないでください。
- 画面へEntityを直接渡す実装を増やさず、既存のBean/Formと`BeanTools`の利用方針に合わせてください。
- 一覧取得ではデータ量増加を考慮し、不要な全件取得、N+1クエリ、Java側だけでの大量データ絞り込みを避けてください。
- フェーズ2では、Issue #2、#3、#4、#6により既存テーブルの変更または履歴データ等の追加が発生する可能性があります。確定したテーブル名、カラム、制約、保持期間は各Issue本文を正とし、実装前のコード調査結果と合わせて決定してください。
- Issue #5のレコメンドは、同一カテゴリと売れ筋情報を基準とします。閲覧履歴など他Issueの未確定な実装へ暗黙に依存させないでください。

# 5. 認証・認可の現状

## 権限値

`Constant`で以下の権限値を使用しています。

| 定数 | 値 | 権限 |
| --- | ---: | --- |
| `AUTH_SYSTEM` | 0 | システム管理者 |
| `AUTH_ADMIN` | 1 | 運用管理者 |
| `AUTH_CLIENT` | 2 | 一般ユーザー |

権限を文字列や別の数値で重複定義せず、既存の定数を使用してください。

## Filter構成

`FilterConfig`では次の順番でFilterを登録しています。

1. `LoginCheckFilter`
2. `ClientAccountCheckFilter`
3. `AdminAccountCheckFilter`
4. `SystemAdminAccountCheckFilter`
5. `CategoryListMakeFilter`

URL判定は主に`URLCheck`へ集約されています。新しいURLを追加した場合は、ゲスト、一般ユーザー、運用管理者、システム管理者の各ロールでFilterの通過条件を確認してください。

画面上のリンクを非表示にするだけでは認可になりません。URLを直接入力した場合やcurlでリクエストした場合も、サーバー側で拒否されなければなりません。

## 6.2 セキュリティおよび不足機能

Issue #1～#3は、セキュリティおよび不足機能に関する実装です。

### Issue #1：管理者権限によるアクセス制御機能

- 運用スタッフ用の管理者と一般ユーザーを分離すること。
- `/admin/**`は適切な管理者権限だけがアクセスできること。
- 一般ユーザー同士で、他ユーザーの会員情報・注文情報等を参照または更新できないこと。
- ゲスト、一般ユーザー、管理者の認証・垂直認可・水平認可をサーバー側で検証すること。
- 画面上のリンクを隠すだけでなく、既存の独自Filterとサーバー側処理でアクセスを拒否すること。

### Issue #2：ログイン失敗時のアカウントロック機能

- 同一アカウントでログインに連続5回失敗した場合、一定時間ログイン不可にすること。
- ロック判定、失敗回数のリセット、ロック解除時刻などの詳細は対象Issueの受け入れ条件に従うこと。
- アカウントの存在有無を不要に外部へ知らせるエラーメッセージを追加しないこと。
- ロック中は認証成功扱いにせず、セッションへ`user`を保存しないこと。

### Issue #3：ログイン履歴管理機能

- ログイン成功時だけ履歴を保存すること。
- ログイン失敗を成功履歴として保存しないこと。
- 会員画面で最新履歴と過去履歴を確認できること。
- 一般ユーザーは自分のログイン履歴だけを参照でき、他ユーザーの履歴をID指定等で参照できないこと。
- 履歴の表示件数、並び順、保存項目、保持方法はIssue本文を正とすること。
- 履歴表示URLを追加する場合は、`LoginCheckFilter`、権限Filter、`URLCheck`の対象範囲を確認すること。

### CRUDとセキュリティの共通確認

- 実装前に対象データのCreate、Read、Update、Deleteを確認すること。
- 論理削除の場合はDeleteではなくUpdateとして扱うこと。
- 更新・削除が管理者限定か、本人限定かを明示すること。
- 退会済みメールアドレスの再登録など、論理削除後の再作成ルールを考慮すること。
- 必須のセキュリティ確認コマンドが通るよう、FilterとURL判定を設計すること。

## 6.3 顧客体験（UX）の向上と販促機能

チームでは、追加要望メールの候補からIssue #4～#6を実装します。

### Issue #4：最近見た商品の表示機能

- ユーザーの商品閲覧履歴を保存すること。
- 商品詳細画面に最近見た商品を表示すること。
- 履歴の対象ユーザー、重複時の扱い、表示件数、並び順、保存期間はIssue本文を正とすること。
- 他ユーザーの閲覧履歴を表示しないこと。
- Issue #5の関連商品とは表示目的と選定ロジックを分離すること。

### Issue #5：関連商品のレコメンド表示機能

- 現在の商品と同一カテゴリの商品を候補とすること。
- 既存の売れ筋情報を利用して関連商品を選定すること。
- 現在表示中の商品、削除済み商品等の除外条件と最大表示件数はIssue本文に従うこと。
- 閲覧履歴へ依存せず、Issue #4が未実装または変更中でも単独で動作できること。
- 候補が存在しない場合でも商品詳細画面を正常表示すること。

### Issue #6：配送希望日指定機能

- 注文手続きで配送希望日を指定できること。
- 配送希望日を注文情報として保存すること。
- 注文確認画面と注文履歴で配送希望日を表示すること。
- 指定可能な日付範囲、必須・任意、未指定時の扱いはIssue本文を正とし、画面側だけでなくサーバー側でも検証すること。
- 注文確定前のセッション情報と、注文確定後のDB保存値を一致させること。

## 6.4 会員向け追加機能

チームでは、Issue #7～#11として、購入後の体験、販促、および注文時の利便性を向上する機能を実装します。

### Issue #7：商品レビュー機能

- 一般ユーザーは、自分の注文詳細画面に表示された購入済み商品の商品名横にある「レビューを書く」ボタンから、レビュー入力画面へ遷移できること。
- レビューは1～5の整数による評価、任意の本文、投稿日を保持すること。評価は必須、本文は未入力を許可し、投稿日はサーバー側で設定すること。
- 商品詳細画面下部に、他の一般ユーザーが投稿したレビューを含む当該商品のレビューを表示すること。
- レビュー投稿時は、セッションのログインユーザーが対象商品を実際に購入済みであることをサーバー側で検証し、注文ID・商品ID等のリクエスト値だけを信用しないこと。
- 1注文商品あたり、または1ユーザー・1商品あたりの投稿可能数、レビューの編集・削除可否、表示順、表示件数はIssue本文を正とすること。
- 削除済み商品、退会済み会員、注文取消等がレビュー表示・投稿可否へ与える影響をIssue本文で確定し、他ユーザーの注文を経由した投稿を拒否すること。
- レビュー用Entityを追加する場合は、注文・会員・商品との参照関係、重複防止制約、必要なインデックスを含むOracle SQLを同じPRへ提出すること。

### Issue #8：お気に入り商品機能

- 一般ユーザー向けサイドバーのカテゴリ検索の下に「お気に入り商品一覧」への導線を追加すること。
- 商品詳細画面の「買い物かごに入れる」ボタンの左側に、お気に入り登録ボタンを追加すること。
- お気に入りはログインユーザー単位で管理し、一覧画面には本人が登録した商品だけを表示すること。
- 同一ユーザーが同一商品を重複登録できないこと。登録済み商品の再登録時の扱い、お気に入り解除方法、一覧の並び順はIssue本文を正とすること。
- 商品IDやユーザーIDを改ざんしても、他ユーザーのお気に入りを参照・追加・削除できないようサーバー側で所有者を検証すること。
- 削除済み商品は新規登録対象および通常の一覧表示から除外し、登録後に商品が削除された場合の扱いはIssue本文に従うこと。
- お気に入り用テーブルを新設し、会員・商品への外部キー、ユーザーと商品の組み合わせに対するUNIQUE制約、必要なインデックス、Sequence、ロールバックを含むOracle SQLを同じPRへ提出すること。

### Issue #9：クーポン機能

- 一般ユーザー向けサイドバーに「マイクーポン」と「クーポンガチャ」への導線を追加すること。
- マイクーポン画面でログインユーザーが保有するクーポンを一覧表示し、会員詳細画面でも同じユーザーの保有クーポンを確認できること。
- クーポンガチャは会員ごとに1日1回利用でき、日次の切替時刻は日本時間の午前7時とすること。利用可否は画面表示やJavaScriptだけでなくサーバー側で判定し、同時リクエストでも1日1回を超えて付与されないこと。
- ガチャの結果は「はずれ」「5％割引クーポン」「10％割引クーポン」とし、各結果の抽選確率、クーポンの有効期限、重複保有可否はIssue本文を正とすること。
- クーポンによる割引は注文代金に対する割合で計算すること。対象金額、端数処理、送料等の扱い、1注文で利用できる枚数、他の割引やIssue #11のポイントとの適用順序はIssue本文で確定すること。
- クーポンの定義、会員への付与、ガチャ実行履歴、注文での利用状態をDBで管理し、使用済みクーポンの再利用や他ユーザーのクーポン利用をサーバー側で拒否すること。
- 注文確定時は、利用可能性の再検証、割引額の計算、クーポンの使用済み更新、注文保存を不整合が生じない単位で処理すること。
- 必要なテーブル、Sequence、外部キー、UNIQUE制約、チェック制約、インデックス、初期クーポンデータ、ロールバックを含むOracle SQLを同じPRへ提出すること。

### Issue #10：お届け先複数登録機能

- 一般ユーザーは会員詳細画面から、自分のお届け先を最大3件まで登録・閲覧できること。
- 注文時のお届け先入力画面では、ログインユーザーの登録済みお届け先から選択でき、初期選択は「アドレス1」とすること。「アドレス1」が存在しない場合の初期選択はIssue本文を正とすること。
- 登録済みお届け先の選択肢の下に「新しい住所に届ける」リンクを設け、今回の注文に使用する新しい住所を入力できること。
- 新しい住所を登録済みお届け先へ保存するか、注文限りとするか、ならびに登録済みお届け先の更新・削除・表示名・並び順はIssue本文を正とすること。
- 最大3件の制限はサーバー側で検証し、同時登録による上限超過も防止すること。セッションのログインユーザーを所有者として扱い、他ユーザーのお届け先をID指定で参照・更新・削除・注文利用できないこと。
- 注文確認画面に選択または入力した住所を表示し、注文確定時には後から登録住所が変更されても注文時の配送先を再現できる形式で`orders`へ保存すること。
- お届け先用Entityを追加する場合は、会員への外部キー、上限・並び順を管理する制約、必要なインデックス、Sequence、ロールバックを含むOracle SQLを同じPRへ提出すること。

### Issue #11：ポイント機能

- 注文確定のたびに、付与対象金額の1％に相当するポイントを注文者へ付与すること。付与対象金額、端数処理、付与時期、注文取消・返品時の扱いはIssue本文を正とすること。
- 1ポイントを1円として注文代金へ利用でき、利用ポイントは100ポイント単位とすること。
- 支払方法選択画面で現在の保有ポイント、利用可能ポイント、利用ポイント入力欄、およびポイント利用による値引額を確認できること。
- 注文確認画面で利用ポイント、ポイント値引額、および今回の注文で付与予定のポイントを表示すること。
- 利用ポイントは0以上、100ポイント単位、保有ポイント以下、かつポイント適用対象金額以下であることをサーバー側で検証すること。不正値や改ざん値をセッションまたは注文へ保存しないこと。
- 注文確定直前に残高と利用条件を再検証し、ポイント消費、注文保存、付与処理を二重実行や残高不整合が生じない単位で処理すること。
- 注文には利用ポイント、ポイント値引額、付与ポイント等の確定値を保存し、後から換算・付与ルールが変わっても注文履歴の金額を再現できること。
- 会員のポイント残高だけでなく、付与・利用・取消を追跡できる履歴データを保持すること。必要なEntity変更と、既存会員・注文データへのバックフィル、制約、インデックス、Sequence、ロールバックを含むOracle SQLを同じPRへ提出すること。

## 6.5 Issue間の境界と並行開発

各機能の詳細仕様、実装方式、変更対象ファイル、Entity・DB変更の有無は、各Issue本文と実装開始時点のコード調査によって確定します。このガイドだけを根拠に変更ファイルを決めつけないでください。

- Issue #1～#11は原則として単独で実装・テストできる構成にすること。
- 他Issueへの依存が必要な場合は、依存先Issue、前提PR、必要なマージ順をIssueとPRに明記すること。
- Issue #2と#3はログイン成功・失敗処理の近接領域を扱うため、認証判定、セッション生成、履歴保存の責務を混同しないこと。
- Issue #4と#5は商品詳細画面へ表示を追加するため、画面上の配置、Model属性名、CSSクラス名を担当者間で調整すること。
- Issue #6は注文入力、確認、確定、履歴表示を横断するため、注文途中のセッション情報と保存後のEntityを区別すること。
- Issue #7と#8は商品詳細画面を変更するため、Issue #4・#5を含め、画面上の配置、Model属性名、CSSクラス名を担当者間で調整すること。
- Issue #9と#11は注文金額の計算および注文確定処理を変更するため、割引の適用順序、端数処理、確定値の保存項目、トランザクション境界を担当者間で合意すること。
- Issue #10はIssue #6と同じ注文入力・確認・確定画面を変更するため、セッション属性名、入力Form、`orders`へ保存する項目を担当者間で調整すること。
- 共通ファイルを変更する場合は差分を最小化し、担当Issueと無関係な整形やリファクタリングを行わないこと。
- 先行PRが`dev`へマージされた場合、後続ブランチは最新`dev`を取り込んでから再テストすること。
- 競合を避ける目的で、未マージの他Issueの機能を先回りして実装しないこと。

変更対象が重複すると判明した時点で、担当者間で着手順とマージ順を調整してください。推測上の変更候補ではなく、各ブランチで実際に変更するファイル一覧をPRへ記載して調整します。

# 7. 開発時の絶対禁止事項

- `pom.xml`の変更を伴う新しいバックエンドライブラリを導入しないこと。
- Spring Securityを導入しないこと。
- 認証・認可を外部ライブラリへ置き換えないこと。
- 新しい業務サービスレイヤを作らないこと。
- ControllerからRepositoryを呼ぶ既存の基本MVC構成を、Issueの範囲外で全面的に変更しないこと。
- 認可をThymeleafの表示制御やJavaScriptだけで実装しないこと。
- ユーザーから送信されたユーザーID・注文IDだけを信用して、他ユーザーのデータを取得・更新しないこと。
- Entityだけ変更して、対応するOracle SQLを提出しない状態にしないこと。
- 既存の論理削除データを安易に物理削除しないこと。
- Issueにない大規模リファクタリング、命名の一括変更、パッケージ再編成を同じPRへ混ぜないこと。
- 文字化けしている既存コメントやメッセージを、機能実装と同時に一括修正しないこと。
- `target/`、`temp/`、IDE設定、ローカルDB設定、生成物をコミットしないこと。
- 研修資料のWord・Excelファイルを、明示的な依頼なしに変更しないこと。
- 他メンバーまたはユーザーの未コミット変更を上書き・削除しないこと。
- `main`または`dev`へ直接機能実装をコミットしないこと。
- エージェント判断だけでPull Requestをマージしないこと。

JavaScriptライブラリは要件上例外として利用可能ですが、可能なら既存機能と標準JavaScriptを優先してください。CDNを使用する場合は、利用目的、URL、バージョン、ライセンス、CDN障害時の影響をPull Requestに記載してください。

# 8. 実装ルール

## 8.1 実装前

1. 対象Issueの概要、制約、受け入れ条件、対象外を確認する。
2. 現在のブランチと作業ツリーを確認する。
3. `dev`の最新状態から`feat#<Issue番号>`ブランチを作成する。
4. 変更候補ファイルを調査し、他機能との競合箇所を洗い出す。
5. 対象機能のCRUD、認証、垂直認可、水平認可への影響を確認する。
6. Entity変更の有無とSQL提出の要否を決める。
7. 既存テストと手動確認手順を確認する。

## 8.2 Java

- クラス名はUpperCamelCase、メソッド・変数名はlowerCamelCase、定数はUPPER_SNAKE_CASEとします。
- 原則としてGoogle Java Styleを参考にしつつ、周辺コードのインデントと構成を優先します。
- マジックナンバーや権限値を直接記述せず、`Constant`または機能内の意味ある定数を使用します。
- 入力値検証はForm、Bean Validation、独自Validatorの既存パターンに合わせます。
- Controllerへ同じ処理を重複実装しない一方、Issue外の大規模抽象化は行いません。
- 例外を握りつぶさず、利用者向け表示とログ出力を分けて考えます。
- 秘密情報、パスワード、セッションID、個人情報をログへ出力しません。

## 8.3 Repository / JPA

- `JpaRepository`と既存の派生クエリまたは`@Query`の書き方に合わせます。
- 論理削除対象では原則として`deleteFlag = Constant.NOT_DELETED`相当の条件を含めます。
- 一般ユーザーのデータ取得では、必要に応じてログインユーザーIDを検索条件に含めます。
- 一覧は可能な限りDB側で絞り込み・並び替え・件数制限します。
- 関連データ取得ではN+1に注意し、JOIN、集約、Pageable等を必要な範囲で使用します。
- JPQLやメソッド名から結果順序が明確になるようにします。
- Oracleで実行できないDB固有構文を使用しません。

## 8.4 EntityとSQL

Entityを追加・変更する場合は、同じPRにOracle用SQLを必ず含めてください。

SQLには、変更内容に応じて以下を含めます。

- `CREATE TABLE`または`ALTER TABLE`
- Sequence作成・変更
- 主キー、外部キー、UNIQUE、NOT NULL、DEFAULT等の制約
- 既存データへの初期値・バックフィル
- 必要なインデックス
- 開発・レビュー用のロールバックSQL
- 必要に応じたテストデータSQL

SQLの保存先がIssueまたはチーム規約で指定されている場合はそれに従います。指定がない場合は、`shared_shop/sql/feat-<Issue番号>.sql`のように機能単位で新規作成し、PRに適用順序を記載してください。

既存データが存在する前提で、NOT NULLカラムを初期値なしで追加しないでください。EntityのJava型、`@Column`名、NULL許可、Sequence名とSQL定義を一致させてください。

## 8.5 Thymeleaf / CSS / JavaScript

- 共通レイアウトと既存CSSクラスを再利用します。
- 表示文言は可能な限り`messages.properties`へ追加し、テンプレートへのハードコードを避けます。
- フォーム送信先、HTTPメソッド、hidden値、バリデーションエラー表示を確認します。
- 画像未登録時は既存の`no_image.jpg`表示パターンを維持します。
- 画面幅が狭い場合でも、文字・ボタン・画像が重ならないようにします。
- JavaScriptを無効化しても、重要な認証・認可・価格計算・入力検証を回避できない設計にします。
- 画面変更時は、PC幅とスマートフォン幅の両方を視覚的に確認します。

## 8.6 文字コードと差分

- ソースファイルはUTF-8として扱います。
- 一部の既存コメントやメッセージには、過去の文字コード変換による文字化けがあります。
- 機能と無関係な文字コード変換、改行コードの全置換、ファイル全体の整形を避け、差分を最小化してください。
- 日本語文言を追加した場合は、ブラウザ表示とプロパティファイルの読み込み結果を確認してください。

# 9. セキュリティ実装ルール

## 認証（AuthN）

- 未ログイン状態でログイン必須URLへアクセスした場合、ログイン画面へリダイレクトすること。
- ログイン成功時だけセッションへ`user`を保存すること。
- ログアウト・退会完了時はセッションを無効化すること。
- 退会済みユーザーはログインできないこと。
- Issue #2のロック中アカウントは、正しいパスワードであってもIssue所定の解除条件を満たすまでログインできないこと。
- Issue #3のログイン履歴は、認証成功が確定した後に保存し、保存処理によって認証結果やセッション生成が不整合にならないこと。

## 垂直認可

- 一般ユーザーは`/admin/`配下へアクセスできないこと。
- 運用管理者とシステム管理者のアクセス可能範囲を既存`URLCheck`とFilterに合わせること。
- 新しい管理URLを追加した場合、全権限で直接URLアクセスを確認すること。

## 水平認可（IDOR対策）

- 一般ユーザーAが一般ユーザーBの注文、会員情報、ログイン履歴、商品閲覧履歴等をID指定で参照・更新・削除できないこと。
- URLやFormのIDではなく、セッションの`UserBean.id`を本人判定の基準にすること。
- Repository検索条件へ所有者IDを含める、または取得直後に所有者を検証すること。
- 研修要件に従い、Filterでも対象URLへのアクセス制御を実装・確認すること。
- 不正時は機密データをレスポンスへ含めず、要件に従ってログイン画面、403、または安全な画面へ遷移させること。

## Filter変更時

- `chain.doFilter()`が二重実行されないこと。
- リダイレクト後に処理を継続しないこと。
- 静的ファイル、ログイン、新規会員登録等の公開URLを誤って遮断しないこと。
- GETだけでなくPOSTの直接実行も確認すること。
- Filterの登録順序を変更する場合は、全ロールへの影響を説明すること。
- URLの部分一致を追加する場合、似た別URLを誤判定しないことをテストすること。

- 論理削除済みデータ
- 未ログイン
- 一般ユーザー

研修テンプレートのセキュリティ別シートを基準に、少なくとも以下をcurlまたは同等手段で確認します。

- エラー表示、0件表示、長い文字列、画像なしを確認する。
- ブラウザコンソールエラーがないことを確認する。

# 11. Gitと並行開発

## ブランチ

- `main`：研修成果の安定版
- `dev`：追加機能の統合先
- `feat#<Issue番号>`：Issue単位の実装ブランチ

原則として1ブランチ・1Issue・1機能とします。

```text
main
 └─ dev
     ├─ feat#1  管理者権限によるアクセス制御機能
     ├─ feat#2  ログイン失敗時のアカウントロック機能
     ├─ feat#3  ログイン履歴管理機能
     ├─ feat#4  最近見た商品の表示機能
     ├─ feat#5  関連商品のレコメンド表示機能
     ├─ feat#6  配送希望日指定機能
     ├─ feat#7  商品レビュー機能
     ├─ feat#8  お気に入り商品機能
     ├─ feat#9  クーポン機能
     ├─ feat#10 お届け先複数登録機能
     └─ feat#11 ポイント機能
```

## 作業ルール

1. 最新の`dev`からfeatureブランチを作成する。
2. Issueの範囲だけを変更する。
3. 小さく意味のある単位でコミットする。
4. 自動テストと手動確認を行う。
5. `feat#<Issue番号>`から`dev`へのPull Requestを作成する。
6. レビュー指摘へ対応し、チーム確認後にマージする。
7. 他のfeatureがマージされたら、次の競合機能着手前に最新`dev`を取り込む。

他ブランチの機能へ強く依存する実装を避けてください。依存が不可避な場合は、前提PR、マージ順、競合ファイルをIssueとPRへ明記してください。

## コミット・PR言語

- エージェントとのセッション内説明は日本語で記述する。
- コミットメッセージとPull Request本文は日本語で記述する。
- クラス名、メソッド名、DB名等の技術識別子は既存の英語表記を維持する。

## Pull Requestに必ず記載する内容

- 対応Issue
- 実装概要
- 仕様上の判断
- 変更ファイル
- Entity・DB変更の有無
- SQLの適用順序とロールバック方法
- 自動テスト結果
- 手動確認結果
- セキュリティ確認結果
- 画面変更時のスクリーンショット
- 既知の制約、未対応事項
- 競合しやすい他機能・ファイル

# 12. エージェントの作業報告ルール

# 13. Definition of Done

以下をすべて満たした場合に、追加機能の実装完了とします。

- Issueの受け入れ条件を満たしている。
- 担当するIssue #1～#11の機能境界を守り、別Issueの未マージ実装へ暗黙に依存していない。
- 必須の認証・認可・CRUD観点を確認している。
